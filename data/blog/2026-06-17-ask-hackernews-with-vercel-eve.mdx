Vercel recently announced [Eve](https://vercel.com/eve), a framework for building
agents. We used it to build a demo,
[Ask HackerNews](https://upstash-hackernews-eve-agent.vercel.app), that answers
questions about HackerNews from real data.

The agent uses retrieval-augmented generation (RAG). Instead of answering from the
model's memory, it queries a search index and bases its answer on the results.
The index is [Upstash Redis Search](https://upstash.com/docs/redis/search/introduction),
and the data is the HackerNews archive that also powers
[HackerNews Trends](https://hackernewstrends.com/).

You can ask it things like "top stories about Rust", "what do people say about
remote work", or "average score of stories vs jobs". The code is on GitHub:
[`upstash/eve-example`](https://github.com/upstash/eve-example).

## The data

The dataset is the same one behind [HackerNews Trends](https://hackernewstrends.com/),
our dashboard for HackerNews topic trends. Stories, comments, Ask/Show HN posts,
jobs and polls all live in a single Upstash Redis Search index named `hn`, along
with their titles, authors, scores and timestamps.

## Upstash Redis Search

[Upstash Redis Search](https://upstash.com/docs/redis/search/introduction) adds
full-text search and secondary indexes to Upstash Redis. You define a schema over
your data and query it over HTTP, with filtering, sorting and aggregations. There
is no separate search service to deploy.

You don't write to the index directly. You store data with normal Redis commands,
and the index picks up keys that match its prefix. The `hn` index is a hash index
over the `hn:` prefix:

```ts
import { Redis } from "@upstash/redis";

const redis = Redis.fromEnv();
const hn = redis.search.index({ name: "hn" });

// Fields: title/text (TEXT), by/type (KEYWORD),
//         score/ndesc/parent (F64), time (DATE)
```

A few query shapes cover what the agent needs:

```ts
// Full-text + filter + sort
await hn.query({
  filter: { $and: [{ title: { $smart: "postgres" } }, { type: { $eq: "story" } }] },
  orderBy: { score: "DESC" },
  limit: 5,
});

// Count without fetching
await hn.count({ filter: { by: { $eq: "pg" } } }); // → { count: 10559 }

// Group items by type
await hn.aggregate({
  aggregations: { by_type: { $terms: { field: "type" } } },
});
```

`$smart` combines fuzzy, phrase and term matching in one operator, which works
well for natural-language input. There are more operators (facets, histograms,
percentiles, highlighting), but the basics handle most questions.

## Letting the model write the queries

We didn't give the agent one fixed "search HackerNews" tool. It has three tools,
`query`, `count`, and `aggregate`, and each one passes a raw Upstash Redis Search
options object to the SDK:

```ts
// agent/tools/query.ts
export default defineTool({
  description: "Run an Upstash Redis Search QUERY against the HackerNews index…",
  inputSchema: z.object({
    filter: z.any().optional(),
    select: z.any().optional(),
    limit: z.number().int().optional(),
    orderBy: z.any().optional(),
    /* … */
  }),
  async execute(options) {
    return hn.query(options); // forwarded verbatim
  },
});
```

The query syntax and the index schema are in the agent's system prompt, so the
model writes its own filters. Ask for the most upvoted stories about Postgres and
it produces:

```jsonc
{
  "filter": { "$and": [
    { "title": { "$smart": "Postgres" } },
    { "type": { "$eq": "story" } }
  ]},
  "select": { "title": true, "score": true, "by": true },
  "orderBy": { "score": "DESC" },
  "limit": 5
}
```

![A grounded answer with the underlying query tool call](/blog/ask-hackernews-with-vercel-eve/query-turn.png)

Aggregations run on the server, so a question like "break down items by type"
returns counts per type without fetching any documents. Because the tools are
generic, new kinds of questions work without new code, and every answer is backed
by real items the model can cite.
